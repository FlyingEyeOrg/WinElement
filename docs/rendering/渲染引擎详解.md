# WinElement 渲染引擎详解

> 对应源码：`include/winelement/rendering/` + `src/rendering/` + `src/platform/win32/d3d11*.cpp`
> 核心组件：RenderCommandList、RenderScene、D3D11DisplayListRenderer、TextEngine

---

## 一、整体架构

### 1.1 渲染管线总览

```
应用层（UIElement 树）
   │ 每帧 paint()
   ▼
阶段 1：录制 ── RenderContext → RenderCommandList（SOA 命令存储）
   │ 提供指纹用于跳过检测
   ▼
阶段 2：场景构建 ── RenderScene（保留式节点树）
   │ 脏区域裁剪 + Compositor 合成分析
   ▼
阶段 3：帧分析 ── RenderFrameGraph（Pass 分组 + 依赖分析）
   │
   ▼
阶段 4：GPU 执行 ── D3D11DisplayListRenderer
   ├─ 脏区域 → dirty clips → render tiles
   ├─ 并行录制（deferred context × N）
   ├─ 批处理 + Draw Call 合并
   └─ Present → 屏幕
```

### 1.2 四个阶段的目标

| 阶段 | 输入 | 输出 | 目标 |
|------|------|------|------|
| **录制** | UIElement::paint() 调用 | RenderCommandList | 把 UI 树转成扁平命令流 |
| **场景构建** | RenderCommandList | RenderScene（节点树） | 重建嵌套结构（Layer/Clip） |
| **帧分析** | RenderScene | RenderFrameGraph | 规划 Pass 顺序 + 并行区域 |
| **GPU 执行** | RenderFrameGraph | 屏幕像素 | 状态排序 + 批处理 + Draw |

## 二、RenderCommandList（命令列表）

### 2.1 设计：SOA（Struct of Arrays）

传统 AOS 存储的问题是命令负载大小不一，如用一个 union 存储所有类型会导致浪费。SOA 把每种命令类型的数据存在独立的 vector 中：

```
opcodes_ (RenderOpcodeRecord)
┌──────────────────────────────────────┐
│ opcode: FillRect, offset: 42, ...    │
│ opcode: DrawText,  offset: 7,  ...   │
│ opcode: FillRect, offset: 43, ...    │
│ opcode: PushLayer, offset: 0,  ...   │
└──────────────────────────────────────┘

独立的负载向量：
fill_rect_payloads_[42, 43, ...]
draw_text_payloads_[7, ...]
push_layer_payloads_[0, ...]
```

**18 个独立负载向量**：每个 `RenderCommandType` 一种，包括 fill_rect、stroke_rect、fill_rounded_rect、draw_text、draw_image、push_clip、push_layer 等。

### 2.2 模板分派

```cpp
template <typename Payload>
[[nodiscard]] const Payload& payload(std::size_t opcode_index) const;

// 使用：
const auto& rect = command_list.payload<FillRectCommand>(i);
// 编译期展开为：fill_rect_payloads_[opcodes_[i].payload_index]
```

### 2.3 命令追加

```cpp
void FillRect(Rect rect, Color color) {
    auto& recorder = context.recorder;
    recorder.append<FillRectCommand>({rect, color});
}

// append 内部：
// 1. 写入对应的 payload vector（fill_rect_payloads_）
// 2. 在 opcodes_ 中追加一条记录
// 3. 更新 bounds（命令列表的边界框）
```

### 2.4 批处理预分析

```cpp
vector<RenderDrawBatch> compute_draw_batches() const;
```

遍历 opcodes，为相邻的兼容命令生成 `RenderDrawBatch`：

```cpp
struct RenderDrawBatch {
    RenderBatchKind kind;         // State | Geometry | Text | Image | Effect
    RenderCommandType type;
    Rect bounds;                  // 该批次所有命令的边界
    uint64_t state_key;           // 混合模式、颜色等
    uint64_t resource_key;        // 纹理 ID
    uint32_t command_count;       // 命令数
};
```

兼容判断由 `draw_batch_key_for(opcode)` 完成——相同 key 的命令可以合并在一个 batch 中，GPU 状态不变。

### 2.5 指纹（Fingerprint）

```cpp
uint64_t fingerprint() const noexcept;
```

对所有 opcodes 做 hash mix，生成一个 64 位指纹。用于：
- **场景构建跳过**：如果指纹和上一帧相同，整个场景构建过程跳过
- **节点复用**：子命令列表指纹不变时复用旧的 PreparedRenderCache

指纹包含：opcode 类型、payload 索引、bounds、类型特定的 payload 指纹（颜色、位置、样式等）。

## 三、RenderScene（保留式场景树）

### 3.1 节点树结构

```
RenderNode (树)
├── kind: Picture | Clip | Layer | Surface
├── fingerprint: uint64_t
├── bounds: Rect
├── transform: Transform2D
├── opacity: float
├── clips_to_bounds: bool
├── children: vector<RenderNode>
│
├── Picture: 包含一个 RenderCommandList（子命令列表）
├── Clip: 包含裁剪区域 + 子节点
├── Layer: 包含变换/透明度 + 子节点（可提升为合成层）
└── Surface: 离屏渲染目标
```

### 3.2 场景构建（update_from_commands）

从扁平的 RenderCommandList 解析出树状的 RenderScene：

```
parse_retained_node(command_list, start_index):
  收集连续的普通命令（FillRect、DrawText 等）→ Picture 节点
  遇到 PushLayer → 递归 parse 到 PopLayer → Layer 节点
  遇到 PushClip  → 递归 parse 到 PopClip  → Clip 节点
  遇到 Save/Restore → 跟踪状态栈
```

### 3.3 节点复用

`reuse_matching_nodes()` 比较新老场景的节点：

```
新节点指纹 == 旧节点指纹 && bounds == bounds && transform == transform
  → 直接复用旧节点的 PreparedRenderCache
  → 不重新计算几何体和字形数据
```

不匹配 → 丢弃旧节点，重新生成。

### 3.4 Picture 缓存

```cpp
class RenderPictureCache {
    LruCache<uint64_t, vector<RenderNode>, 128> cache_;
    // key: 子命令列表的指纹
    // value: 缓存的渲染节点
};
```

128 项 LRU 缓存。当一个 Picture 的指纹命中，跳过整个子树的场景重建。

## 四、Compsoitor（合成层提升）

### 4.1 什么情况下提升为合成层

Compositor 分析场景树中的 Layer 节点，判断哪些可以提升为 DirectComposition 独立表面：

| 条件 | 说明 |
|------|------|
| **面积 >= 4096 DIP²** | 太小的层不值得单独合成 |
| **边界有限** | 不能是 NaN 或无限 |
| **类型** | StableLayer（跨帧稳定）、AnimatedOpacity、AnimatedTransform |

### 4.2 稳定层识别

`stable_layer_id()` 通过 hash 跨帧识别同一层：

```cpp
hash = command_count + opcode_index + bucketed_position
// position 用 16px 桶量化：x & ~0xF, y & ~0xF
// 允许微小的位置偏移保留同一 ID
```

这样 DirectComposition 可以跨帧复用合成表面，不需要重建。

### 4.3 合成帧构建

```cpp
class CompositorFrameBuilder {
    // 构建 CompositorFrame
    // 包含：帧 ID、清理颜色、DirtyRegion、RenderScene、提升的图层列表
};
```

## 五、帧分析（Frame Graph）

### 5.1 Pass 类型

```cpp
enum class RenderFramePassKind {
    State,      // 混合/深度状态变更
    Geometry,   // FillRect、FillRoundedRect 等
    Text,       // 文字渲染
    Image,      // 图片渲染
    Effect,     // 阴影、特效
    Composite,  // 合成层合并
    Present,    // 显示
};
```

### 5.2 Pass 分组

`build_render_frame_graph()` 遍历 opcodes，把相同类型的连续命令合并为 Pass：

```
command list:  FillRect → FillRect → DrawText → DrawText → FillImage
                    ↓           ↓          ↓          ↓          ↓
passes:      [ Geometry Pass ] [ Text Pass ] [ Image Pass ]
                  合并           合并          单独
```

Pass 组的依赖分析决定了哪些 Pass 可以并行录制，哪些有顺序依赖。

### 5.3 缓存

`NodeGraphCache` 是线程本地缓存（64 项），key 为 `(node_fingerprint + node_kind + bounds)`。当场景节点指纹不变时直接复用上次的帧分析结果。

## 六、D3D11 渲染器

### 6.1 架构

```
D3D11DisplayListRenderer
├── 管道状态（4 种混合/模板组合）
├── 顶点批处理器（batch_vertices, flush_batch）
├── 状态栈（clip/transform/opacity/stencil_depth）
├── 字形图集（glyph_atlas_*, 2048×2048 RGBA8）
├── 文本渲染（DWrite → atlas → GPU）
├── 几何管线（flatten → tessellate → fill/stroke）
├── 脏区域裁剪（dirty_clips → render tiles → scissor）
├── 并行录制（deferred contexts × N）
└── 缓存系统（4 级缓存）
```

### 6.2 管道变体（4 种）

| 变体 | 混合模式 | 模板 | 用途 |
|------|----------|------|------|
| AlphaBlend | (ONE, INV_SRC_ALPHA) | 关闭 | 普通几何/图片 |
| SubpixelText | (ONE, INV_SRC1_COLOR) | 关闭 | ClearType 文本 |
| StencilAlphaBlend | (ONE, INV_SRC_ALPHA) | 测试 | 裁剪后的几何 |
| StencilSubpixelText | (ONE, INV_SRC1_COLOR) | 测试 | 裁剪后的文本 |

### 6.3 顶点批处理

`submit_vertices()` 是核心路径：

```
submit_vertices(顶点数据, 纹理, 纹理模式, 模板深度):
  如果当前批次兼容（纹理/模式/深度一致 && 不超过 16383 顶点）:
    追加到 batch_vertices
  否则:
    flush_batch()
    
flush_batch():
  cached_pipeline_for(纹理模式, 模板深度) → 选择管道变体
  bind_blend_state()
  bind_depth_stencil_state()
  draw_vertices_now()

draw_vertices_now():
  更新常量缓冲区（窗口尺寸、DPI）
  绑定着色器资源视图（纹理）
  Map vertex_buffer_ → 拷贝顶点 → Unmap
  Draw(顶点数, 0)
```

**state tracking**：跟踪最近绑定的 blend state、depth stencil state、shader resource，避免重复调用 D3D11 的 Set 方法。

### 6.4 状态栈

```cpp
struct DrawState {
    Rect clip_rect;           // 当前裁剪矩形
    Transform2D transform;    // 当前变换
    float opacity;            // 当前透明度
    int stencil_depth;        // 当前模板深度（嵌套裁剪层数）
};

vector<DrawState> state_stack;  // push_clip/pop_clip 时入栈/出栈
```

常见操作：
- `push_clip(rect)` → 裁剪矩形与当前裁剪求交，入栈
- `push_layer(transform, opacity)` → 叠加变换到当前变换，入栈
- `Save/Restore` → 完整状态入栈/出栈

### 6.5 并行录制

```
render(frame):
  如果是简单帧（无 layer、少命令）:
    单线程录制 → FinishCommandList → ExecuteCommandList
  
  如果是复杂帧:
    analyze_render_frame() → 拆分成 RenderWorkItem
    ┌─ Clear（清屏，barrier）
    ├─ Node[0] + Tile[0]（线程 0）
    ├─ Node[0] + Tile[1]（线程 1）
    ├─ Node[1] + Tile[0]（线程 2）
    └─ 合并：submit_recorded_work_items()
          按 sort_key 排序后依次 ExecuteCommandList
```

工作线程从 `render_thread_pool_` 获取，录制的 deferred context 从 `worker_recorder_pool_` 获取。最大并行为 2 个（超过收益递减）。

## 七、字形图集

### 7.1 图集结构

| 参数 | 值 |
|------|-----|
| 尺寸 | 2048 × 2048 |
| 格式 | RGBA8（32 位/像素） |
| 大小 | **16 MB** |
| 打包方式 | 逐行排列，光标推进 |
| 间距 | 每字形 1px padding（防止线性采样时相邻字形渗色） |

### 7.2 字形光栅化流程

```
draw_text("Hello")
  → DWrite CreateTextLayout → 获取字形索引和位置
  → 对每个字形：
    → 查 glyph_atlas_entries_ 哈希表
      → 命中：直接取 UV 坐标 → 生成纹理四边形
      → 未命中：rasterize_glyph_coverage()
        → IDWriteGlyphRunAnalysis 创建
        → 尝试 ClearType 渲染（3 通道）
        → 回退到 Aliased 渲染（1 通道）
        → 转换为 RGBA8 + padding
        → pack 到图集中（光标推进）
        → 标记脏区域
        → 记录到 glyph_atlas_entries_
    → 提交 6 个顶点（纹理四边形）到批处理器
```

### 7.3 图集上传

```
upload_glyph_atlas_if_dirty():
  脏区域 < 50% 图集大小:
    → UpdateSubresource(dirty_box)  只更新脏的矩形区域
  脏区域 >= 50%:
    → UpdateSubresource(null)        全量更新
  清空脏标志
```

### 7.4 图集重置

图集满了 → `reset_glyph_atlas()`：
- 清空 `glyph_atlas_entries_` 哈希表
- 重置光标到 (0, 0)
- `glyph_atlas_generation_`++（所有缓存的 glyph run 失效）
- 从头开始填充

重置是昂贵的——下一帧所有文字都走"rasterize + pack + upload"路径。

### 7.5 字形回退

如果字形未在图集中找到（空间不足或光栅化失败），走**轮廓回退**：

```
glyph_outline_geometry():
  → IDWriteFontFace::GetGlyphRunOutline() → 获取路径几何
  → 通过 fill_geometry() 作为矢量路径渲染
  → 比图集采样慢得多
```

## 八、文本渲染管线

### 8.1 DWrite 文本布局（CPU 侧）

```
TextEngine::layout_text("Hello", TextStyle):
  1. normalize_text("Hello") → NFC 标准化
  2. CreateTextLayout("Hello", max_width, max_height)
  3. GetLineMetrics() → TextLine 数组
  4. GetClusterMetrics() → TextCluster 数组
  5. CapturingTextRenderer::DrawGlyphRun() → 捕获字形索引/位置/字体
  6. synthesize_missing_clusters_from_glyphs() → 填补 DWrite 未报告的字簇
  7. 构建 CaretStop 数组（用于光标/选区）
  8. 计算可见字节范围（视口裁剪）
  9. 缓存结果到 LRU（64 项）
```

### 8.2 GPU 端渲染

```
draw_text_layout(layout, position, color):
  第一步：遍历所有字形
    → 查 glyph_atlas_entries_ 哈希表
    → 生成纹理四边形顶点（位置 + UV）
    → 提交到批处理器（TextureSamplingMode::RgbSubpixelCoverage）
  
  第二步：处理未在图集中找到的字形
    → glyph_outline_geometry()
    → fill_geometry() 作为路径渲染
  
  缓存：text_glyph_run_cache_（48 项 LRU）
    key: layout_fingerprint + prepared_glyphs_signature + atlas_generation
    图集重置时自动失效
```

### 8.3 文本预准备

在并行录制前，`prepare_text_resources_for_scene()` 遍历所有可见的 DrawText 命令，确保所有字形已存在于图集中。这步串行执行，减少了并行录制期间各自的字形光栅化开销。

## 九、几何体管线

### 9.1 路径扁平化（Flatten）

复杂曲线 → 线段近似：

```
二次贝塞尔：递归细分，弦长误差 < 0.5px
三次贝塞尔：递归细分，弦长误差 < 0.5px
椭圆弧：SVG 弧参数 → 端点转中心点 → 分段线性近似
椭圆段落：adaptive_ellipse_segments(bounds) → 按周长计算段数
```

### 9.2 多边形三角化（Tessellation）

`tessellate_geometry_fill()` — 扫描线活动边算法：

```
1. 收集所有 Y 值（顶点 Y + 边交点 Y）
    → 边-边交点检测：O(E²) — 已知性能问题
2. 对每个 Y 带：
   a. 构建活动边列表（按 X 排序）
   b. 按 winding number 填充
   c. 输出三角形顶点
3. 支持 EvenOdd / NonZero 填充规则
```

### 9.3 轮廓清理

```
collinear_point_removal():
  检测三点共线 → 移除中间点

redundant_point_deduplication():
  移除连续重复的坐标点
```

### 9.4 描边渲染

`stroke_polyline()` — 完整描边管线：

```
1. Dash 处理：遍历线段，精确定位 dash 起止点
2. Join 处理：Miter（斜接）/ Bevel（斜面）/ Round（圆角）
3. Cap 处理：Butt（平头）/ Square（方头）/ Round（圆头）/ Triangle（三角）
4. AA fringe：每条边的 4 个横向偏移
   → outer_aa, solid, solid, outer_aa（覆盖率抗锯齿）
5. 连续网格优化：所有 join 同为 miter 或 round 时生成统一网格
```

### 9.5 圆角矩形优化

圆角矩形不走通用几何管线——`fill_rounded_rect()` 直接用缓存的 1/4 圆弧点列表生成三角网格。角半径 clamp 到矩形尺寸的一半。

### 9.6 阴影渲染

`draw_box_shadow()` — 多层近似高斯模糊：

```
1. 基础填充：18% alpha
2. 外层 16 层同心扩大矩形
3. 每层 alpha = (1.0 - t * 0.75) / steps
4. 不需要 blur shader 的 GPU 友好方案
```

## 十、脏区域驱动管线

### 10.1 脏区域优化

```
DirtyRegionOptimizeOptions:
  max_rects = 96      // 最多保留的脏矩形数
  merge_slop = 0.0    // 合并容差
  scanline_merge = true

optimize():
  1. 移除空矩形
  2. 按面积降序排序
  3. 合并重叠矩形（union）
  4. 如覆盖 >65% 面积 → 合为整屏（全屏脏）
  5. 如超出 max_rects → 查找并合并最近的两个
```

### 10.2 脏区域 → 渲染裁剪

```cpp
dirty_clips_from_region(dirty_region):
  for each dirty rect:
    D3D11RenderDirtyClip clip;
    clip.device_clip = rect;       // 用于 scissor rect
    clip.cull_clip = rect + 1px;   // 用于裁剪测试（保守扩展）
```

### 10.3 可见性裁剪

```cpp
collect_visible_opcode_indices(command_list, dirty_clips):
  for each opcode:
    if opcode.bounds 与 任一 dirty_clip.cull_clip 相交:
      → 可见，加入渲染列表
    else:
      → 跳过（GPU 不会处理此命令）
```

### 10.4 渲染 Tile

```cpp
build_render_tiles(dirty_clips, target_size):
  // 按目标尺寸计算 tile 网格
  // 每个 tile 对应一个 dirty_clip
  // 完全被单个 dirty_clip 覆盖的 tile 跳过 per-opcode 相交测试
  // 用于并行录制分配
```

### 10.5 清屏优化

```cpp
clear_dirty_region(dirty_clips):
  if 单个 dirty_clip 覆盖全屏:
    → ClearRenderTargetView（最快）
  else:
    → 对每个 dirty_clip 设置 RSSetScissorRects
    → 用清理颜色的四边形填充
```

## 十一、缓存系统

### 11.1 四级渲染缓存（RenderPlanCacheState）

| 级别 | 最大项数 | key | 用途 |
|------|----------|-----|------|
| FrameGraph | 16 | 场景指纹 | 帧分析结果 |
| VisibilityTile | 48 | tile key | 每个 tile 的可见命令索引 |
| PassMetrics | 16 | pass key | 每 pass 可见命令数 |
| RenderPlan | 16 | full plan key | 完整渲染计划（tiles + 命令索引） |

所有级别 LRU 淘汰，60 帧空闲后清除。

### 11.2 PreparedRenderCache（几何体/字形预处理）

| 缓存 | key | 存储内容 |
|------|-----|----------|
| geometry_flatten | geometry_hash | 扁平化的图形数据 |
| geometry_fill | canonical_key | 三角化顶点 + bounds |
| geometry_stroke | stroke_key | 描边预处理数据 |
| text_glyph | text_fingerprint | 预光栅化的字形覆盖位图 |

默认 128 项，通过 LRU 风格的 deque 淘汰。

### 11.3 变换几何体缓存

`transformed_geometry_fill_cache_`（64 项）：key = PreparedFill 指针 + transform，缓存在给定变换下已经处理的几何体填充数据。

## 十二、纹理上传与缓存

### 12.1 双通道上传队列

```
RenderResourceUploadQueue:
├── HighFrequencySmall lane（高频小上传）
│   └── 小纹理、部分更新
└── LowFrequencyLarge lane（低频大上传）
    └── 大图片、整图更新
```

每个通道有独立的 mutex，大上传不阻塞小上传。

### 12.2 Upload Action

| Action | 含义 |
|--------|------|
| Upload | 从像素数据创建新纹理 |
| Retain | 增加引用计数 |
| Release | 减少引用计数（为 0 时删除） |
| Discard | 立即删除 |

### 12.3 D3D11 纹理缓存

`D3D11RenderResourceCache` 管理 GPU 纹理：

- 使用 `D3D11_USAGE_IMMUTABLE` + `D3D11_SUBRESOURCE_DATA` 初始化（一旦创建不可更新，效率最高）
- 引用计数管理
- 设备丢失后通过 `ResourceUploadReplayCache` 重放所有上传

## 十三、设备丢失处理

### 13.1 检测

```
D3D11CompositionSurface::render():
  Present1() → DXGI_ERROR_DEVICE_REMOVED / DXGI_ERROR_DEVICE_RESET
  → 返回 RenderResult::DeviceLost
```

### 13.2 恢复流程

```
1. handle_device_lost():
   → discard() 释放所有设备依赖资源（swapchain、RTV、renderer、resource cache）
   → device_->recreate() 新建 D3D11 设备
   → 创建新的 swapchain、RTV、renderer

2. WindowRenderWorker 检测到 DeviceLost:
   → ResourceUploadReplayCache 重放所有资源上传
   → 重新渲染当前帧
   → 如果再次失败 → RenderResult::Failed
```

### 13.3 ResourceUploadReplayCache

- 按 ID 存储每个资源的上传负载
- 设备重建后按顺序重放：Retain → Upload/Discard
- 上限 16×2N+1 次 pass（内存有界）

## 十四、DirectComposition 集成

### 14.1 架构

```
D3D11CompositionSurface（swapchain）
  │ CreateSwapChainForComposition
  ▼
DirectCompositionBridge（IDCompositionDevice）
  ├── CreateVisual → 可视化树
  ├── SetContent(swapchain) → 绑定合成表面
  └── Commit() → 提交到 DComp 系统
```

### 14.2 合成层提升

对于被 Compositor 选中提升的 Layer，DirectComposition 在合成器线程上处理变换/透明度/位置，不占 GPU 渲染时间。这是你的框架拖拽丝滑的原因之一——窗口的合成由 DComp 驱动，不经过 D3D11 渲染管线。

## 十五、资源生命周期

### 15.1 全部 D3D11 资源的创建和销毁

| 资源 | 创建时机 | 销毁时机 |
|------|----------|----------|
| Device | 窗口创建 | 窗口销毁 |
| SwapChain | 窗口创建 | 窗口销毁/设备丢失 |
| VertexBuffer | 每帧（Map WRITE_DISCARD） | 自动（每帧新数据覆盖） |
| ConstantBuffer | 设备创建 → 固定存在 | 设备销毁 |
| BlendState × 4 | 设备创建 | 设备销毁 |
| StencilTexture | 按需（首次需要模板时） | 120 帧空闲后释放 |
| GlyphAtlasTexture | 设备创建 | 设备丢失/销毁 |
| ImageTextures | 图片加载时 | 引用为 0 或设备丢失 |
| DeferredContexts | 按需（并行录制时） | 归还到池 |

### 15.2 缓存逐出策略

| 缓存 | 逐出触发 | 方式 |
|------|----------|------|
| PreparedRenderCache | 超过 128 项 | LRU deque 头部淘汰 |
| TextLayout | 超过 64 项 | LRU 淘汰 |
| FontFamilyCache | 超过 512 项 | clear() 全清 |
| GlyphRunCache | 48 项 | LRU + generation 失效 |
| RenderPlanCache | 空闲 60 帧 | prune_unused() |
| StencilTarget | 空闲 120 帧 | release_stale_stencil_target() |
| IdleResource | 空闲 3 秒 | trim_idle_resources() |

## 十六、完整出帧流程

```
WM_PAINT
  │
  ├─ 1. BeginPaint → 获取 rcPaint
  ├─ 2. DirtyRegion.add(rcPaint)
  │
  ├─ 3. post_ui_frame(request)
  │     │
  │     ├─ UI 工作者线程开始
  │     ├─ 4. 布局 → calculate_layout(LayoutConstraints)
  │     ├─ 5. 动画 → Storyboard::tick()
  │     │
  │     ├─ 6. 录制 → UIElement::paint()
  │     │        → RenderContext::fill_rect() 等
  │     │        → append 到 RenderCommandList
  │     │
  │     ├─ 7. 场景构建 → RenderScene::update_from_commands()
  │     │        → 指纹比对 → 跳过或重建
  │     │        → 节点复用
  │     │
  │     ├─ 8. Compositor 分析 → 选择合成层
  │     │
  │     └─ 9. render_worker_->render(frame)
  │           │
  │           ├─ 渲染工作者线程开始
  │           ├─ 10. 预准备 → glyph_atlas_entry() 预光栅化字形
  │           ├─ 11. 帧分析 → build_render_frame_graph()
  │           ├─ 12. 脏区域 → dirty_clips → render tiles
  │           ├─ 13. 并行/串行录制 → deferred contexts
  │           │     ├─ Clear → 清屏（脏区域）
  │           │     ├─ 遍历可见命令 → 生成顶点 + 批处理
  │           │     └─ Flush batch → Draw call
  │           ├─ 14. 合并 ExecuteCommandList
  │           ├─ 15. Present1(1, 0) → 显示
  │           └─ 16. 检查 DeviceLost → 恢复
  │
  └─ 17. EndPaint → 完成
```
