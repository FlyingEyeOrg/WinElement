# 08 — 延迟上下文与多线程录制

> 对应渲染器概念：`primary_deferred_context_`、`render_thread_pool_`

## 问题

D3D11 的 `ID3D11DeviceContext` 默认是**立即模式**——你调 `Draw()` 时，命令立刻被发送到 GPU。

```
主线程：
  context->Draw(...) → 立即发送到 GPU → GPU 开始执行
  context->Draw(...) → 立即发送到 GPU → GPU 继续执行
  context->Draw(...) → 立即发送到 GPU → GPU 继续执行
```

问题是：**CPU 在等待 GPU 响应时不能做其他事**。而且你只有一个 Context，不能从多个线程同时往同一个 Context 中发命令。

## 延迟上下文的解决方案

D3D11 引入了**延迟上下文（Deferred Context）**。它像一个"命令录像机"——你调 `Draw()` 时它**不发送到 GPU**，只是记录下来。

```
线程 1 的延迟上下文：              线程 2 的延迟上下文：
  deferred_ctx1->Draw(A)               deferred_ctx2->Draw(B)
  deferred_ctx1->Draw(C)               deferred_ctx2->Draw(D)
       ↓                                    ↓
  生成 CommandList1                     生成 CommandList2
       ↓                                    ↓
              主线程立即上下文：
                context->ExecuteCommandList(list1)
                context->ExecuteCommandList(list2)
                → GPU 执行 A, C, B, D
```

这样就可以**用多个线程并行录制命令**，然后合并提交。

## 创建延迟上下文

```cpp
ID3D11DeviceContext* deferred_ctx;
device->CreateDeferredContext(0, &deferred_ctx);
```

注意：延迟上下文是从 **Device** 创建的，不是从 Context。

## 使用延迟上下文

```cpp
// 线程 1：
ID3D11DeviceContext* ctx1 = deferred_contexts[0];
// 像普通 context 一样使用
ctx1->IASetVertexBuffers(...);
ctx1->VSSetShader(...);
ctx1->Draw(6, 0);

// 录制完成后，生成命令列表
ID3D11CommandList* cmd_list1;
ctx1->FinishCommandList(FALSE, &cmd_list1);

// 线程 2 做同样的事...
// 线程 3 做同样的事...

// 主线程合并
immediate_context->ExecuteCommandList(cmd_list1, FALSE);
immediate_context->ExecuteCommandList(cmd_list2, FALSE);
immediate_context->ExecuteCommandList(cmd_list3, FALSE);
```

## 限制

延迟上下文不是完全自由的：
1. **不能访问 SwapChain**：延迟上下文里不能调 `Present`
2. **不能 Map/Unmap**：延迟上下文里不能更新常量缓冲区
3. **状态隔离**：每个延迟上下文有自己独立的状态，共享资源（纹理、缓冲区）需要注意同步
4. **Create* 操作不从延迟上下文走**：创建纹理、缓冲区还是通过 Device

你的渲染器实践中：**资源上传在 CPU 线程做，命令录制在工作线程做，合并提交在主 Context 做**。

## 在你的渲染器中

```cpp
// d3d11_display_list_renderer.cpp
// 使用 render_thread_pool_ 并行处理可见区域

// 把脏区域切成多个 tile
auto tiles = split_dirty_region(dirty_region);

// 每个 tile 分配到一个线程
for (auto& tile : tiles) {
    thread_pool.submit([this, tile]() {
        // 每个线程有自己的 deferred context
        auto& deferred_ctx = thread_local_context();

        // 在这个 tile 上录制命令
        render_tile(tile, deferred_ctx);

        // 生成 command list
        ID3D11CommandList* cmd_list;
        deferred_ctx->FinishCommandList(FALSE, &cmd_list);
        return cmd_list;
    });
}

// 主线程合并
for (auto& result : thread_pool.wait_all()) {
    immediate_context->ExecuteCommandList(result, FALSE);
}

// 提交 GPU
swapchain->Present(1, 0);
```

`primary_deferred_context_` 不是线程池里的那几个延迟上下文——它是渲染器本身的"主延迟上下文"，当不启用线程池时用于命令录制，单线程时也用它。

## 性能收益

```
单线程录制：                         多线程录制（4 线程）：
                                    ┌─────────────────┐
主线程：                            │ tile 0: 线程 0   │
  Draw(所有命令)                     ├─────────────────┤
  ← 约 10ms →                      │ tile 1: 线程 1   │
                                    ├─────────────────┤
                                    │ tile 2: 线程 2   │
                                    ├─────────────────┤
                                    │ tile 3: 线程 3   │
                                    └─────────────────┘
                                     ← 约 3ms →
                                    └──────── 合并 < 1ms
                                    总共约 4ms
```

对于非常多的绘制命令（比如你的 mixed 场景 321 draw calls），多线程录制可以显著减少 CPU 侧的帧时间。

## 注意

延迟上下文**不是免费的**：每个 `FinishCommandList` 生成的命令列表需要在主线程上 `ExecuteCommandList`。如果 tile 太少（比如只有 2 个 tile），多线程的开销可能抵消收益。你的渲染器只在脏区域足够大时才启用并行录制。
