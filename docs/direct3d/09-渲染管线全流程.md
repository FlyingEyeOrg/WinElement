# 09 — 渲染管线全流程

> 整合所有概念：从 DirtyRegion 到 Present

这一章把前面 8 章的知识串起来，展示一个完整的帧是如何从 UI 事件到屏幕像素的。

## 一帧的完整路径

```
用户点击按钮 → set_text("hello")
  │
  ▼
[UIElement 层]
  invalidate_paint()
  → 标记脏标志
  → InvalidateRect(hwnd, &rect)  // 告诉 Windows 这块区域需要重绘
  │
  ▼
[Windows 消息循环]
  WM_PAINT
  → BeginPaint → 拿到 paint.rcPaint（脏区域）
  → DirtyRegion.add(rcPaint)     // 记录这一帧哪些像素变了
  │
  ├─ 普通帧 → post_ui_frame()    // 异步，丢到工作者线程
  └─ 拖拽帧 → prepare_ui_frame() // 同步，当前线程执行
  │
  ▼
[UI 工作者线程 / 当前线程]
  prepare_ui_frame(request)
  │
  ├─ 1. 布局阶段
  │   → UIElement::calculate_layout()
  │   → Yoga 重新计算所有脏元素的布局
  │   → 更新 LayoutElement::frame()
  │
  ├─ 2. 动画阶段
  │   → Storyboard::tick()
  │   → 更新正在运行的动画属性
  │
  ├─ 3. 录制阶段
  │   → UIElement::paint()  遍历 UI 树，调用 RenderContext
  │     → fill_rect / draw_text / fill_rounded_rect ...
  │     → append 到 RenderCommandList
  │   → 生成 329 条命令（mixed 场景）
  │
  ├─ 4. 场景构建阶段
  │   → RenderScene::update_from_commands()
  │   → 解析 RenderCommandList，构建 RenderNode 树
  │   → 指纹对比（如果没变化，跳过解析）
  │   → 复用上一帧的 PreparedRenderCache
  │
  ├─ 5. 裁剪阶段
  │   → DirtyRegion 里的脏区域 → 裁剪列表
  │   → 用 DirtyRegion 裁剪 RenderScene
  │   → 跳过不相交的子树和命令
  │
  ├─ 6. 渲染提交阶段
  │   → 打包成 RenderFrame
  │   → render_worker_->render(frame)
  │
  ▼
[渲染工作者线程]
  render(frame)
  │
  ├─ 7. 准备阶段
  │   → 更新 constant_buffer_（窗口尺寸、DPI）
  │   → 确保字型图集 glyph_atlas_texture_
  │   → 预布局文本（prepare_text_resources）
  │
  ├─ 8. 命令执行阶段（D3D11）
  │   → 遍历 RenderScene 的 RenderNode
  │   → 对每个可见命令：
  │     │
  │     │  DrawText("Hello"):
  │     │  ├─ 查 glyph_atlas_entries_ → 找到每个字形的 UV 坐标
  │     │  ├─ 生成 6 个顶点（2 个三角形组成四边形）
  │     │  ├─ 设置 blend_state_ = subpixel_text_blend_state_
  │     │  ├─ 设置 stencil_state_ = disabled
  │     │  └─ Draw(6, 0)
  │     │
  │     │  FillRoundedRect(圆角矩形):
  │     │  ├─ 设置 stencil_state_ = increment → 画圆角到模板
  │     │  ├─ 设置 stencil_state_ = test → 画矩形内容
  │     │  ├─ 设置 stencil_state_ = decrement → 清理模板
  │     │  ├─ 设置 blend_state_ = blend_state_
  │     │  └─ Draw(6, 0) × 3（画 3 次）
  │     │
  │     │  FillRect(矩形):
  │     │  ├─ 生成 6 个顶点
  │     │  ├─ 设置 blend_state_ = blend_state_
  │     │  └─ Draw(6, 0)
  │     │
  │     │  DrawImage(图片):
  │     │  ├─ 生成 6 个顶点（带 UV 坐标）
  │     │  ├─ 绑定图片纹理到 PSSetShaderResources
  │     │  └─ Draw(6, 0)
  │
  ├─ 9. 批处理优化
  │   → merge_batches():
  │     把连续的相同状态的 Draw 合并
  │     → FillRect(红) + FillRect(红) + FillRect(红)
  │        = Draw(18, 0)  一次画 18 个顶点
  │
  ├─ 10. Present
  │    → swapchain->Present(1, 0)
  │    → GPU 把后缓冲交换到前缓冲
  │    → 屏幕刷新 → 用户看到新的一帧
  │
  ▼
[完成]
  回到消息循环，等待下一帧
```

## 你的管线在每帧中的开销分布

以 mixed 场景（329 commands / 321 draw calls / 100.2ms render_ms）为例：

```
时间轴：
0ms        20ms        40ms        60ms        80ms        100ms
├──────────┼───────────┼───────────┼───────────┼───────────┤
│          │           │           │           │           │
[布局]     [录制]      [场景构建]  [DWrite    [GPU 渲染   [Present]
  ~1ms     ~15ms       ~15ms      布局]        ~10ms      ~1ms
                                    ~55ms
```

**瓶颈分析**：
- `DWrite 布局 ~55ms`：TextLayout 缓存 12 slot，thrashing 严重
- `录制 ~15ms` + `场景构建 ~15ms`：每帧全量重建
- `GPU 渲染 ~10ms`：实际 GPU 干活的时间其实是够的

## 对比你的其他场景

```
dense_dirty (579 cmds, 0.93ms render):
[布局 ~1ms][录制 ~3ms][场景 ~2ms][GPU ~1ms][Present ~1ms]
                                    ↑ 纯几何，不用 DWrite，极快

text (14 cmds, 11.4ms render):
[布局 ~1ms][录制 ~2ms][场景 ~2ms][DWrite ~8ms][GPU ~1ms][Present ~1ms]
                                    ↑ 只有文本，DWrite 占比高

svg (50 cmds, 17.8ms render):
[布局 ~1ms][录制 ~3ms][场景 ~2ms][几何 ~12ms][GPU ~2ms][Present ~1ms]
                                    ↑ O(N²) 边-边交叉
```

## 优化的方向

理解了全流程后，优化的方向就很清晰了：

| 优化 | 影响的阶段 | 预期收益 |
|------|-----------|----------|
| TextLayout 缓存 12 → 128+ | DWrite 布局 | 55ms → 5ms |
| 消除双重文本布局 | DWrite 布局 | 消除重复 DWrite |
| 几何体 O(N²) → 空间分区 | 几何预处理 | 12ms → 3ms |
| 增量 Scene 更新 | 场景构建 | 15ms → 3ms |
| 脏区域在 UI 层裁剪 | 录制 | 跳过不可见控件的录制 |

## 总结

你的渲染器是一个**两阶段、四层**的架构：

```
CPU 阶段：
  第 1 层：UIElement 树 ─ 逻辑交互、状态管理
  第 2 层：RenderCommandList ─ 录制绘制命令
  第 3 层：RenderScene ─ 场景树 + 裁剪 + 缓存

GPU 阶段：
  第 4 层：D3D11 ─ 顶点缓冲 → 着色器 → 混合 → 模板 → 屏幕
```

理解了这个全流程，你就理解了 `d3d11_display_list_renderer.cpp` 中每一行代码存在的理由。
